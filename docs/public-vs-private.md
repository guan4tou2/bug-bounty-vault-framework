# Public vs Private Vault Boundary

The framework is designed to start as a Public Seed and evolve into a Private Vault. The structure stays the same; the data and private adapters grow locally.

| Concern | Public Seed | Private Vault |
|---|---|---|
| Purpose | Reusable architecture, workflow, templates, and safety rules | Your operational security research system |
| Targets | `_example` only | Real target folders and target-specific notes |
| Findings | Placeholder examples and schemas | Confirmed Findings, Attempts, Submissions, and FORMs |
| workspace | Empty scaffold, `.gitignored` | Local scratch for logs, PoC, screenshots, scanner output, and firmware work |
| LLM Wiki | Seed Patterns, Playbooks, and Knowledge Capture rules | Self-updating LLM Wiki built from your own sessions |
| Skills and agents | Platform-neutral skills and generic agents | Optional private adapters and target-specific automation |
| Report/Form output | Generic Submission and FORM templates | Downstream-channel formats generated by private adapters |
| bbflow | Contract, scope guard, output format, and scanner config seeds | Installed tools, real scopes, run manifests, candidates, and triage history |
| Obsidian | Recommended folder layout and plugin settings | Your complete research workspace and graph |
| Git visibility | Public-safe by default | Usually private, because it contains target-specific and operational data |

## Boundary Rules

Use the public seed for stable structure:

- session lifecycle
- target and finding schemas
- scope validation
- Knowledge Capture gate
- platform-neutral report writing
- scanner configuration examples
- LLM skill and agent contracts

Use the private vault for operational data:

- target-specific findings
- private notes and hypotheses
- downstream channel adapters
- raw evidence and screenshots
- workspace logs and scan results
- private prompts or workflow tweaks that reveal methodology tied to real targets

## Update Direction

Promote generic improvements from the Private Vault back to the Public Seed only after sanitization.

Good upstream candidates:

- a safer session checklist
- a better scope validation rule
- a reusable note template
- a generic evidence readiness check
- a platform-neutral workflow clarification

Keep private:

- target-specific exploit chains
- program-specific wording
- channel-specific form field mappings
- real scanner output
- any finding, screenshot, account, host, token, or log line
